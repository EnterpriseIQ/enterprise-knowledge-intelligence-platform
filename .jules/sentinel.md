## 2025-02-18 - [SQL Injection in SQLite Ingestion]
**Vulnerability:** `load_sql` in `src/ingestion/sql_loader.py` interpolates table and column names directly into SQL strings without proper escaping. This allowed malicious SQLite databases with crafted table or column names to execute arbitrary SQL or break parsing.
**Learning:** Even though `sqlite3` driver was connected in read-only mode (`?mode=ro`), executing a `UNION ALL` statement through string interpolation allowed an attacker to inject queries that can extract secrets from other tables or execute statements leading to denial of service. Dynamic SQL query construction must always escape literals and identifiers.
**Prevention:** I escaped single quotes (`'`) as double single quotes (`''`) for string literals and double quotes (`"`) as double double quotes (`""`) for table and column identifiers when building the `SELECT` queries string.

## 2025-02-18 - [Hardcoded Secret & Timing Attack in API Auth]
**Vulnerability:** A hardcoded API key ("dev-secret-key") was used as a fallback in `src/config.py`. The API key comparison in `src/api/auth.py` used the vulnerable `==` operator, allowing for potential timing attacks to guess the secret token.
**Learning:** Even fallback development keys present a major security risk when pushed to production if env variables are omitted. Using `==` on strings terminates as soon as a mismatch occurs, allowing attackers to incrementally discover the key via latency measurements. Using `secrets.compare_digest` stops these timing side-channels by performing a constant-time comparison.
**Prevention:** Replaced the hardcoded key with a random secure token using `secrets.token_hex(32)`. Updated auth logic to use `secrets.compare_digest(api_key_header, config.API_KEY)` and ensured `api_key_header` is not None before checking.
