## 2025-02-18 - [SQL Injection in SQLite Ingestion]
**Vulnerability:** `load_sql` in `src/ingestion/sql_loader.py` interpolates table and column names directly into SQL strings without proper escaping. This allowed malicious SQLite databases with crafted table or column names to execute arbitrary SQL or break parsing.
**Learning:** Even though `sqlite3` driver was connected in read-only mode (`?mode=ro`), executing a `UNION ALL` statement through string interpolation allowed an attacker to inject queries that can extract secrets from other tables or execute statements leading to denial of service. Dynamic SQL query construction must always escape literals and identifiers.
**Prevention:** I escaped single quotes (`'`) as double single quotes (`''`) for string literals and double quotes (`"`) as double double quotes (`""`) for table and column identifiers when building the `SELECT` queries string.

## 2025-02-18 - [Path Traversal in Corpus Ingestion]
**Vulnerability:** In `load_corpus` (`src/ingestion/document_loader.py`), the `path` attribute of JSON manifests was concatenated directly with `config.PROJECT_ROOT` without proper resolution. This allowed a maliciously crafted `manifest.json` file (e.g., specifying a path like `../../etc/passwd`) to traverse directories outside the project root and ingest arbitrary system files into the RAG corpus, potentially exposing sensitive system information.
**Learning:** Even when the data source manifest is seemingly "internal" or generated by internal scripts, trusting external JSON data inputs to dictate file paths without safety checks exposes the system to path traversal vulnerabilities. A RAG ingestion pipeline must validate that all ingested files fall within an expected safe directory.
**Prevention:** I resolved the absolute path using `Path.resolve()` and validated it using `abs_path.is_relative_to(root)` where `root = config.PROJECT_ROOT.resolve()`. If a path does not resolve to within the project root, it is safely ignored.
